# ADR-0011: Operational-to-Ledger Integration (Financial Integrity)

## Status
Accepted (2026-07-14). P8.5 remediation for FIN-01 / FIN-02.

## Context
An independent audit (P8) found that revenue generated by the operational domains
(restaurant, bar/liquor, rooms) was **never posted to the General Ledger**. The
accounting auto-posting engine (`accountingService.postAutoPosting` and the
`postRestaurantSale` / `postBarSale` / `postHotelCharge` helpers) existed and was
verified to work, but operational services never invoked it. Bar sales additionally
had no GST/TOT computation (FIN-02). This broke the core invariant: *every business
transaction must become an immutable accounting transaction.*

## Decision
Introduce a single integration seam, `apps/backend/src/domains/accounting/ledger-integration.ts`:

1. **`postOperationalToLedger(source, entry)`** — builds an `AutoPostingEntry` and
   calls `accountingService.postAutoPosting`. On success returns the journal id.
2. **Durable failure queue** — on GL failure, the entry is persisted to a new
   `ledger_post_failures` table (auditable + retryable) so a COA misconfiguration
   never silently drops revenue or breaks the business operation.
3. **`retryLedgerFailures()`** — drains unresolved queue entries once the ledger
   config is corrected.

Operational call sites:
- `restaurant.service.ts:recordPayment` (settled sale) → `restaurant_sale`
- `rooms.service.ts:postPayment` (settled folio) → `hotel_folio_charge`
- `liquor.service.ts:completeSale` → `bar_sale` (also computes Kerala TOT, FIN-02)

## Design
```
Operational service → postOperationalToLedger(source, entry)
        │
        ├─ success → accountingService.postAutoPosting → journal_entries (GL)
        │
        └─ failure → INSERT ledger_post_failures (resolved=0)
                              ↑ retryLedgerFailures() drains on recovery
```

## Trade-offs
- **+** Single, auditable integration point; failures are never lost.
- **+** Non-blocking: a GL fault does not abort the business transaction.
- **-** Adds a background retry responsibility (covered by PERF-09 scheduler + CI).

## Migration
- `schema.sql` adds `ledger_post_failures` and `metric_counters` (idempotent).
- `db.ts` idempotently adds `users.force_password_change` and the
  `gst_registers(created_at)` index (PERF-07).

## Rollback
- Remove the three operational call sites; delete `ledger-integration.ts`.
- No schema rollback required (additive tables/columns).

## Verification
- `tests/financial-integrity.test.ts` asserts: success posts to GL, failure is
  durably queued, and `retryLedgerFailures` drains the queue.
- `tsc --noEmit` clean; ESLint 0 errors.
