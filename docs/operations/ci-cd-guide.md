## Pipeline Overview

```
Code Push ─▶ Lint/Typecheck ─▶ Test ─▶ Build ─▶ Scan ─▶ Push Image ─▶ Deploy (gated)
```

All pipelines share the same stage contract so GitHub and GitLab produce identical artifacts.

> **Note:** Production deploy is **never** automatic — it requires the manual gate from [Release Engineering](release-engineering-guide.md#deployment-gates).

---

## GitHub Actions

Workflows under `.github/workflows/`:

```yaml
name: ci
on: { push: { branches: [main] }, pull_request: {} }
jobs:
  build-test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: npm }
      - run: npm ci
      - run: npm run lint && npm run ts:check
      - run: npm test
      - run: npm run build --workspace=apps/backend
  docker:
    needs: build-test
    runs-on: ubuntu-latest
    steps:
      - uses: docker/build-push-action@v6
        with:
          context: apps/backend
          file: apps/backend/Dockerfile.prod
          push: ${{ github.ref == 'refs/heads/main' }}
          tags: ${{ env.REGISTRY }}/deepa-bms-backend:${{ github.sha }}
```

## GitLab CI

Pipeline defined in `.gitlab-ci.yml` (root). Mirrors GitHub stages:

```yaml
stages: [lint, test, build, scan, deploy]
build-backend:
  stage: build
  script:
    - cd apps/backend && npm ci && npm run build
docker-build:
  stage: build
  script:
    - docker build -f apps/backend/Dockerfile.prod -t $CI_REGISTRY/deepa-bms-backend:$CI_COMMIT_SHA apps/backend
```

> **Note:** Keep both providers' pipelines in sync via a shared stage matrix to avoid drift.

## Local CI

Run the full gate locally before pushing:

```bash
npm run lint
npm run ts:check
npm test
npm run build --workspace=apps/backend
docker build -f apps/backend/Dockerfile.prod -t deepa-bms-backend:local apps/backend
```

> **Warning:** CI may pass locally but fail in CI due to secret/env differences. Always check the pipeline, not just your laptop.

## Build Artifacts

- `apps/backend/dist/` — compiled JS (from `tsc`).
- Docker image `deepa-bms-backend:<sha|version>`.
- OpenAPI spec `docs/api/openapi.yaml`.
- SBOM (generated by scan stage).

## Container Registry

- Push to `ghcr.io/<org>/deepa-bms-backend` (GitHub) or `$CI_REGISTRY` (GitLab).
- Tags: `git sha` for immutability; `vX.Y.Z` for releases; `latest` discouraged in prod pins.
- Pull secrets referenced in K8s/Helm.

> **Note:** Pin production to an explicit version tag, never `latest` — reproducibility matters for [Rollback](release-engineering-guide.md#rollback).

## Environments

| Env | Trigger | Deploy | Gate |
|-----|---------|--------|------|
| dev | PR | auto | none |
| staging | merge to main | auto | CI green |
| production | tag `vX.Y.Z` | manual | release mgr + IC |

Environment-specific values via overlays (`k8s/overlays/*`) or Helm values files.

## Troubleshooting

| Issue | Cause | Fix |
|-------|-------|-----|
| Cache miss slow builds | wrong `cache` key | set node cache = `npm` |
| Image push 403 | bad registry login | re-issue `REGISTRY_*` secret |
| Flaky test | shared state | isolate; use test DB |
| Scan fails build | CRIT vuln | patch base image / dep |
| Drift between GH/GitLab | desynced config | align stage matrix |
